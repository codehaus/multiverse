Multiverse 0.3

Multiverse 0.3 was released at 19 December 2009 afer more than a year of hard working (read almost every bit of spare time I had). This is a list of the most important features of the 0.3 release:
  1. Basic instrumentation so that normal POJO's are able to participate in the Multiverse. This can be done through the @TransactionalObject and @TransactionalMethod annotation, without being forced to extend from some basic class and cripple the design or to introduce awkward code to access fields of transactional objects.
  2. Tracking update transactions: is able to do updates and to participate in blocking operations.
  3. Non tracking readonly transaction: causes less stress on the transaction, so much cheaper than an update transaction. But it isn't allowed to participate in blocking operations and it also is more subject to read conflicts.
  4. Optimization for final fields of TransactionalObjects. Final fields are completely ignored by Multiverse, so there is no overhead on using as many as final fields as possible. If an object
  5. Retry: one of the basic thread notifcation constructs that make it possible to create transactional blocking operations like a BlockingQueue.put.
  6. OrElse: one of the basic thread notification constructs that make it possible to try an alternative path of a blocking operation fails.
  7. TransactionTemplate: a Template method implementation that can be used to create an transactional executed piece of code. The advantage of using a TransactionTemplate is that it provides the same transaction management logic the bytecode-instrumentation provides.
  8. TransactionalLinkedList: a general purpose transactional LinkedList that can be used as:
    1. java.util.Queue
    2. java.util.Deque
    3. java.util.concurrent.BlockingQueue
    4. java.util.concurrent.BlockingDeque
    5. java.util.List
  9. TransactionalRef: a transactional reference to Objects. Can be seen as the transactional version of the java.util.concurrent.atomic.AtomicRef. There also is an TransactionalRef implementation available that doesn't rely on a Java agent. The Akka project from Jonas Boner is an in Scala written distributed actors system that makes use of this functionality.
  10. STM framework: At the moment Multiverse contains a single STM implementation called the AlphaStm which is a general purpose stm implementation. But it certainly is possible to create your own stm implementation and reuse as much as the Multiverse components as needed with having the option to build eveything yourself.
  11. Transaction lifecycle tasks: to make it possible to execute logic just before or after aborts and commits. This is useful to start or clean up non transactional resources.
  12. Basic Scala integration.
  13. The possibility to exclude fields from being transactional using the @Exclude annotation.

Known limitations

  1. Java agent has no support for a transactional object subclassing.
  2. The TransactionalLinkedList is very non concurrent because it causes a lot of conflicts on its internal fields (like the head/tail/size). This means that takes and puts can't execute concurrent and that takes can't happen concurrent and the same goes for takes. This will be solved in the future (perhaps using Transactional Boosting).
  3. No protection against the writeskew problem; an isolation problem. This will be solved in Multiverse 0.4
