Multiverse 0.8

The features that are planned for the 0.8 release are not cast in stone atm, but the following is on the radar:
  1. Abort for: atm a transaction will always abort for any throwable,
  2. Spring support: to make Multiverse easy to integrate with Spring based applications
  3. JTA support:
  4. TransactionalList/TransactionalDeque
  5. Transactional Hashmap/HashSet
  6. Contention Management
  7. Lock acquire policy: atm a configurable amount of spinning is used, but for lock that are acquired for a longer period, spinning is very inefficient. So blocking locks needed and should be done through a lock acquire policy. This is needed to make it possible to use the STM as a more traditional locking solutions
  8. Read/write locks:
  9. Better alignment with Gigaspaces. I use Gigaspaces at work in combination with traditional concurrency control, but I want to experiment with aligning Multiverse with more traditional lock based solutions like GigaSpaces so the traditional lock based concurrency control can be removed.
  10. Profiler
  11. Instrumentation for Java
  12. tryprivatize/tryensure
Improved performance:
  1. instead of doing 'expensive' arrive/depart operations on reads to provide read consistency, for very short transaction it probably is cheaper to do a full conflict scan instead of expensive cas operations for the arrive/depart
  2. less overhead on transactions. I already experimented with transaction implementations where the need for pooling tranlocals is removed since the transaction itself will act as the pool.
  3. removal of volatile writes:
