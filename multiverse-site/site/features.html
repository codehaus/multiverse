Features

  1. Independent of Java language: so it can be used in other languages running on the JVM like Scala or Groovy. Specially for environments that don't want to rely on instrumentation, a direct api to the stm, the transaction and reference factories are provided. This api doesn't depend on a specific Multiverse STM implementation.
  2. Object and field level granularity: the main STM implementation, called the AlphaStm is able to deal with object and field level granularity.
  3. Only transactional access possible: prevents having to deal with accessing transactional objects without a transaction.
  4. Different transaction modes:
    • read tracking update transaction: the work horse that is able to do writes and participate in blocking transactions.
    • non read tracking update transaction: consumes less resources than the read tracking version, but can't participate in blocking transactions or prevent write skew problems.
    • read tracking readonly transaction: the standard transaction implementation for readonly transaction. It is perfect serialized but can't participate in blocking transactions.
    • non read tracking readonly transaction: consumes more resources than the non read tracking version, but is able to participate in blocking transactions.
  5. Transaction lifecycle listener: so that compensating or deferred actions can be executed when a transaction aborts or commits.
  6. Optional write skew prevention: makes is possible to create truly serialized transactions.
  7. Blocking transactions:
    • retry primitive:
    • orelse primitive:
    • interruptible transactions:
    • non blocking transactions:
  8. Transactional references and primitives
    • TransactionalReference: makes it possible to work with managed references. This looks a lot like the approach other STM implementations take like Clojure.
    • Transactional primitive: like the TransactionalInteger and TransactionalBoolean that behave like transactional references but without creating a lot of small wrapper objects.
  9. 2 Phase commit: to make it possible to combine transactions executing on different transactional resources, or even for combining the commit/abort atomic on different transactions on the same stm.
  10. Learning Stm: that learns by runtime transaction analysis and chooses better performing or behaving transactions. Atm it learns about when a transaction should be an update, when it needs read tracking and which type of transaction is the best fit for a certain situation (cheap transaction implementations for small transactions, and more expensive implementations for bigger transactions).
  11. No special JVM needed: Multiverse runs on a standard Java 6+ JVM. Every night all the tests are executed in the newest JVM's of Sun(Oracle), IBM and the OpenJDK.
  12. Not messing other classes: Multiverse will only instrument the classes that are explicitly defined as needing instrumentation. So Multiverse will not mess with your other classes (like those of the JDK).

Planned features

Of course there are also a lot of features in the pipeline. The most important ones are:
  1. Persistence: for making committed transactions durable. This should be mostly transparent to the developer.
  2. Distributed STM: so load can be spread out over a cluster of machines and it also is useful for realizing failover.
  3. Contention management: to have more control on liveness issues like livelocking and starvation.
  4. More collections: like HashMap, TreeMap, SkipList, Tries etc
  5. Improved performance: there is a lot of room for performance improvements by optimizing the bytecode.
