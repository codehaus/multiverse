Multiverse 0.6

Multiverse 0.6 release currently is under development and expected around the end of June 2010. The main focus will not be on adding new features, but on improving scalability, performance and stability.

The expected features are:

  1. Profiling support for transaction and transactional object level behavior. Information will be made available by writing a report to file. In the future more advanced profiling mechanisms will be added, but this should shed some light in the darkness.
  2. New bytecode optimizations:
    1. speculative inlining of transactions: Removing the creation of the transaction object and inlining it completely in simple methods, should give a big performance boost.
    2. transactional objects as arguments: Once the tranlocal is retrieved it can be stored for later use and can be added as extra arguments by widening the methods in the instrumentation. This reduces unwanted access to the transaction and improves the chance for the JIT to do its magic.
    3. removing junk instructions: atm some swaps and duplications of stackframe elements are needed to access the owning transactional object it cause some bytecode noise. Some of this noise will be removed for the 0.6 release so that the JIT has a better chance to do its magic.
  3. Transaction reuse: instead of creating a new transaction every time a transactional method is executed, an old one can be reset and reused.
  4. Better scalable clock. Multiverse 0.5 uses a shared (by all transactions of the same STM) AtomicLong as clock that needs to be incremented by every update transaction (that has updates of course). In 0.5 the tick to the clock is optimized in such a way that concurrent ticks don't need to lead to unique timestamp. (this already removes a lot of pressure on the memory bus). But the 0.6 release is going to have a configurable 'sync to clock' property that relaxes the updating of central clock. This should make the STM more scalable for unshared data structures.
  5. New transactional collections:
    1. TransactionalTreeMap
    2. TransactionalTreeSet
  6. Groovy integration.
  7. Compatible with Scala/Akka. In principle nothing special needs to be done for a class generated by Scala compiler or by the Java compiler to be instrumented; from a bytecode perspective they should be the same. But there always are small exceptions that cause problems. The goal is to make Multiverse Scala/Akka compatible.
  8. Continuous performance integration environment. It should become easy to create benchmark, store results and compare them over time (to do trans analysis). This is needed for the Multiverse project itself to make sure that the performance doesn't degrade.
  9. Bugshaker: a javaagent that increase the chance of concurrency problems to happen by adding random delays at critical sections in the system or user code (probably using some kind of annotation).
